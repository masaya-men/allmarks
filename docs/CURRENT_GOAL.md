# 次セッションのゴール — Fable で残タスクを洗い出し、安価なモデルでも実装できる完璧な計画書を作る

## ★セッションの始め方（ユーザー確定）

1. **Fable を使う**。残タスクを全部洗い出し、**安価なモデル（haiku 等）でもそのまま実装できるレベルの計画書**を書く。
2. 実装はその計画書ができてから。まず計画。

計画書の粒度の基準は s185 の [plan](superpowers/plans/2026-07-10-mobile-share-bottom-nav.md)。実コード・実ファイルパス・実テストコードまで埋まっていて、実装者が判断を迫られない状態にする。s185 の実測では、その粒度なら haiku が単独でタスクを完遂できた（判断が要る配線タスクだけ opus）。

## s185 の実機フィードバック（この4件が起点）

- **(N-56) ★★スマホで共有画像が作成されない（致命・ローンチブロッカー）** — 合成環境（Playwright 390×844）では 5/5 緑だったが実機で失敗。**設計上、撮影が失敗しても共有リンクは作られる**ので「静かに壊れたまま成功に見える」。まず症状の粒度をユーザーに確認する（プレビューが出ない／OG が既定カード／CREATE が error）。原因候補と診断手順は TODO.md N-56。
- **(N-58) ★スマホでもコラージュさせたい** — s185 spec §2.1 で「並べる段は出さない」をユーザー自身が承認したが、実機で触って**表現できないことが受け入れられないと判明**＝**決定を撤回**。「簡素でもコラージュしたい。表現の場なのでスマホでもきちんと表現させたい」。
- **(N-57) スマホのボードに背景タイトルが出ていない** — s185 のスコープ外だった（N-51 の残り）。`BoardBackgroundTypography` の `!isMobile` ゲートを外すだけ。
- **(N-59) スマホでも列数・余白を簡易的に変えたい** — 離散的な選択肢（列 2/3/4・余白 小/中/大）で。`MOBILE_LAYOUT` は今ベタ書き定数。

## 洗い出しに含めるべき既知の残タスク

- **(N-54)** グリッド/クロスハッチの交点だけ濃くなる — 盤面は CSS の重ねグラデーション2枚、受け取り画面/OG は SVG 1描画。**また盤面と共有リンクが食い違っている**。候補＝盤面も同じ SVG data-URI を使う（dom-to-image のグラデ片方向落ちも同時に解消）。`useMemo` 必須。
- **(N-50)** タブレットの作法 — 744〜1180px は PC と同一描画で主要操作が全て 27px。規則は確立済（**大きさは入力で決める・並べ方は幅で決める**）。
- **(N-55)** 撮影成功後もコラージュがシートの裏で触れる（N-58 を実装すると消える）。
- **(N-53)** e2e の腐り — **実態は master 単体で 58本中 30本が落ちる**（s185 で A/B 実測）。`Test timeout` 20件・`VersionError (9)<(16)` 13件。**回帰検出網が半分死んでいる**。
- **(N-28)** 拡張の Pinterest 保存ボタン連動（別枠）。
- リリース段取り: 束C 13言語＋規約 → 束D 公開素材 → 束E 総仕上げ・公開（`docs/private/2026-07-08-release-runway-plan.md`）。

## s185 で出荷したもの（`allmarks.app` 反映済・merge `b9c43511`）

**N-49 スマホから SHARE**（ボトムナビ再設計込み）。tsc0 / vitest 2246 / クリーンビルド / opus 全ブランチレビュー READY TO MERGE。

- ボトムナビ＝`TAG / THEME / SHARE / CORNERS / MORE`（MOTION は MORE パネルへ）
- 撮影＝画面に内接する**中央の 1.91:1 の帯**に自動配置して `.outerFrame` を `fit:'cover'` で撮る。`computeCoverRect` が中央を切るので帯と一致。**黒帯ゼロ・レプリカゼロ**
- 結果シートで `SHARE`（OS の共有シートに画像＋リンク）/ `COPY LINK` / `DONE`
- 回転ノブを触りで出せるように（先に `data-no-capture` を付けてから）
- **レビューが設計の穴を2つ潰した**: ①横長画面で帯≠切り出し ②`fit:'cover'` を誰も固定していなかった（`contain` に戻しても全テスト緑だった）→ 黒帯検出テストを追加

## 絶対に守ること（恒久ルール）

- **ボードの見た目を変えたら、受け取り画面（`/s/<id>`）も必ず確認する。** s183・s184 と同じ事故を繰り返している。
- **撮影は実機でしか検証できない。** s185 は Playwright で 5/5 緑だったが実機で画像ができなかった（N-56）。
- `CardsLayer` に `isMobile` を渡してはいけない（`×` とタグピルが消える）。スクロール解放は `lockCardScroll`。
- カードのタップ選択は `setPointerCapture` を使うので **Playwright では駆動できない**。`SELECT ALL` 経由で確認する。
- **`fit:'cover'` を勝手に `'contain'` に戻さない。** `tests/e2e/mobile-share.spec.ts` の黒帯検出テストが守っている。
- Playwright の `tail` は**失敗リスト**であって実行リストではない。要約行は ANSI 付きなので `sed` で剥がしてから数える。`rtk npx` は壊れる（素の `npx` を使う）。
- 目視確認はユーザーに振る。tsc / vitest / e2e と「測定でしか証明できないもの」だけ自分でやる。
